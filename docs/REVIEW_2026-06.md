## Critical (5)

### MC1 ‚Äî Orphaned WAL/SHM after atomic rename of new `archive.sqlite`
`ios/Sources/Archiver/ArchiveWriter.swift:233-284` ‚Äî `openOrCreateDB` enables `PRAGMA journal_mode=WAL` on the `.tmp` file then `moveItem`s only the main file. Sibling `archive.sqlite.tmp-wal`/`.tmp-shm` are left on disk; the freshly reopened `archive.sqlite` looks for `archive.sqlite-wal` (not `.tmp-wal`) and gets a fresh empty WAL. Any DDL/schema_migrations row not auto-checkpointed pre-close is silently lost ‚Äî bundle is corrupt (no chats/messages/attachments tables).

### MC2 ‚Äî `ArchiveLock.acquire` is not `O_EXCL`-atomic
`ios/Sources/Archiver/ArchiveLock.swift:41-66` ‚Äî sequence is `fileExists ‚Üí read PID ‚Üí removeItem ‚Üí write(atomically:true)`. `atomically:true` clobbers any pre-existing file. Two concurrent acquirers can both pass the stale-PID check, both `removeItem`, both `write` ‚Äî both end up with `ownedByMe = true`. Defeats the lock entirely.

### MC3 ‚Äî Snapshot directories leak GB-scale per failed run
`ios/Sources/Archiver/SourceDBSnapshotter.swift:72-135`, `ios/Sources/Archiver/CreateArchiveCoordinator.swift:73-145` ‚Äî every call creates `~/.imessage-archiver/work/snapshot-<UUID>/chat.db` via `VACUUM INTO`. Nothing deletes the directory ‚Äî not on success, not on failure, not on `cancel()`. Weekly run of a 30 GB chat.db = 30 GB/week of dead disk.

### IC1 ‚Äî `TarReader.extract` length cap of 2 GiB is too loose; caller-dependent enforcement
`ios/Sources/Persistence/TarReader.swift:49-83` ‚Äî the 2 GiB ceiling is way larger than any real attachment. Crafted `tar_length` of 2 GiB forces a 2 GiB allocation. `AttachmentCache` has a 100 MB gate but `Thumbnailer` and any future caller bypass it. Defense in depth: cap inside `TarReader.extract` itself.

### IC2 ‚Äî `AttachmentCache.url(for:)` overwrites symlinks at the destination
`ios/Sources/Persistence/AttachmentCache.swift:70-110` ‚Äî containment check is on the destination path string, not on what's actually at that path. `try data.write(to: cachedURL, options: .atomic)` follows a symlink. iOS sandbox bounds the blast radius; on macOS (App Sandbox disabled per PR #29) a swapped-in symlink can redirect attachment bytes to a write target inside the user's home.

## High (16)

### MH1 ‚Äî `NSUnarchiver.decodeObject()` raises ObjC exceptions that abort the process
`ios/Sources/Archiver/AttributedBodyDecoder.swift:107-124` ‚Äî Swift `try` cannot catch ObjC exceptions. A single corrupt typedstream row kills the whole archive run mid-flight. Need an `@try`/`@catch` ObjC shim.

### MH2 ‚Äî `cancel()` races a second `start()` with the in-flight task
`ios/Sources/Archiver/CreateArchiveCoordinator.swift:54-69` ‚Äî `cancel()` sets `task = nil` synchronously without awaiting. User clicks "Archive" again, `start()` sees `task == nil`, spins up second run. Two runs race for the lock; with MC2 broken, both proceed.

### MH3 ‚Äî Manifest write window where `manifest.json` is missing
`ios/Sources/Archiver/ArchiveWriter.swift:528-571` ‚Äî `write tmp ‚Üí removeItem(manifest) ‚Üí moveItem(tmp ‚Üí manifest)`. Crash between `removeItem` and `moveItem` leaves bundle with no manifest. Must use `replaceItemAt` (APFS-atomic).

### MH4 ‚Äî `messages_fts` declared external-content but populated as standalone
`ios/Sources/Archiver/ArchiveWriter.swift:94-100, 395-399` ‚Äî DDL says `content='messages'` but inserts skip rowid pin. SQLite allocates auto-rowids in the FTS shadow tables that don't relate to `messages.rowid`. Any reader that runs the standard FTS5 `'rebuild'` corruption-recovery will replace shadow content from the live messages table and re-key by `messages.rowid`, potentially corrupting search.

### MH5 ‚Äî SQLite `file:` URI built without percent-encoding
`SourceDBSnapshotter.swift:111-128`, `SourceDBReader.swift:62-65`, `ArchiveMerger.swift:50-54`, `ArchiveVerifier.swift:60-62` ‚Äî `"file:\(path)?mode=ro&immutable=1"` with no encoding. Path with `?`/`#`/`&`/space splits the URI at the wrong `?`. Worst case: mid-string `mode=rwc` substring re-parses and SQLite opens the source chat.db writable ‚Äî direct P0 invariant violation.

### MH6 ‚Äî `attachment_guid` PRIMARY KEY drops N:M joins (DEFERRED to its own PR ‚Äî schema change)
`ios/Sources/Archiver/ArchiveWriter.swift:64-75, 409-462` ‚Äî when one attachment guid is referenced by N messages (forwarded photo, repeat sticker), only the first survives. iOS reader queries `WHERE attachments.message_guid = ?` and shows nothing for messages 2..N. Fix needs `message_attachments` link table + schema_version bump + migration plan for the existing 23 GB archive. Tracked separately for user sign-off.

### MH7 ‚Äî TarWriter truncates UTF-8 mid-codepoint
`ios/Sources/Archiver/TarWriter.swift:60-69, 254-260` ‚Äî `usePAX` triggers at `> 100` UTF-8 bytes. When false (‚â§100 bytes), `writeAsciiZ` slices the first 99 bytes which may land mid-multibyte. Worse: when PAX is used, `String(entryName.prefix(100))` is 100 *graphemes* (potentially far more than 100 bytes), then `writeAsciiZ` truncates again ‚Äî same risk.

### MH8 ‚Äî `chat.db` columns decoded with bare `Int64?` ‚Äî cross-version brittleness
`ios/Sources/Archiver/SourceDBReader.swift:217-264` ‚Äî `is_from_me`, `cache_has_attachments`, `associated_message_type` decoded as `Int64?` with `?? 0`. Older macOS versions store these as BOOL/INTEGER variants; GRDB type-mismatch throws on the whole `messages(in:)` query and `run()` has no per-chat catch ‚Äî silently skips that chat or halts mid-archive.

### IH1 ‚Äî `iCloudCoordinator.handleReady` reads manifest without `NSFileCoordinator`
`ios/Sources/Persistence/iCloudCoordinator.swift:206-226`, `ios/Sources/Models/ArchiveManifest.swift:14-17` ‚Äî bundle URL goes `.ready` and the very next thing is `Data(contentsOf: manifestURL)` without coordinator wrapping. If iCloud has the dir placeholder but not the JSON downloaded, returns partial/empty/throws. ArchiveReader's sqlite open is properly wrapped (PR #30); manifest read should follow same pattern.

### IH2 ‚Äî `ArchiveManifest.load` silently defaults `schema_version` to 1 on parse failure
`ios/Sources/Models/ArchiveManifest.swift:25` ‚Äî `raw["schema_version"] as? Int ?? 1`. Malformed manifest (missing, wrong type, JSON corrupt) treated as v1, defeats PR #16 schema-version refusal. Must use required-key decoding or fail loudly with `ArchiveError.archiveCorrupt`.

### IH3 ‚Äî `Reaction.init(from:)` plus outer `try? ‚Ä¶ ?? []` ‚Äî one bad reaction wipes all reactions on a message
`ios/Sources/Models/Message.swift:60-66`, `ios/Sources/Persistence/ArchiveReader.swift:343` ‚Äî outer decode is all-or-nothing on the whole array. Must decode element-by-element with `try?` per entry so a single bad reaction drops only itself.

### IH4 ‚Äî `messages(in:limit:)` accepts `limit <= 0` and silently drops the LIMIT clause
`ios/Sources/Persistence/ArchiveReader.swift:269-298` ‚Äî a 100k-message conversation returns all rows into a single `[Message]` on the MainActor consumer. Clamp to e.g. `1...1000` at the top of both `search` and `messages`.

### IH5 ‚Äî `NSMetadataQuery` accessed after MainActor hop, racing the query's own queue
`ios/Sources/Persistence/iCloudCoordinator.swift:113-194` ‚Äî handler hops to `@MainActor` then calls `query.disableUpdates()`/`result(at:)`. NSMetadataQuery is documented as not thread-safe; `disable/enable` is meant for the notification callback synchronous bracketing. Set `query.operationQueue = .main` at construction so notifications land on the main queue directly.

### IH6 ‚Äî `ArchiveReader.openDatabase` percent-encodes path with `.urlPathAllowed` which still allows `?`/`#`
`ios/Sources/Persistence/ArchiveReader.swift:94-97` ‚Äî same root cause as MH5 but in the reader. A path containing `?`/`#` corrupts the SQLite URI. Use `URLComponents` instead.

### IH7 ‚Äî `ThreadView` year-picker task never cancelled on disappear
`ios/Sources/Views/ThreadView.swift:140-145` ‚Äî `Task { @MainActor in availableYears = ... }` is unstored; `onDisappear { loadTask?.cancel() }` cancels a separate field. Year-picker keeps churning post-disappear, writes to a `@State` of a dismantled view.

### IH8 ‚Äî `SearchView.debouncedSearch` only checks `Task.isCancelled` *before* the await
`ios/Sources/Views/SearchView.swift:36-49` ‚Äî expensive `reader.search` await has no cancellation guard. Fast-typing user gets stale results overwriting fresh ones. Add `guard !Task.isCancelled else { return }` after the await.
